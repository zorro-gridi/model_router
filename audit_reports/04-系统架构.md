## 4.1 Prompt Complexity Classifier

**设计要求**: 冷启动时对 prompt 做轻量分类，判断高复杂度信号，给出初始模型建议和先验。实现方式可以是规则匹配、关键词+模板、轻量 LLM 评估、规则与小模型混合。

**实现状态**: ✅ 对齐

| 检查点                 | 状态 | 证据                                                                                |
| ---------------------- | ---- | ----------------------------------------------------------------------------------- |
| LLM 轻量分类器         | ✅   | `llm_classifier.py:classify()` — 单次 API 调用完成分类                              |
| 规则匹配兜底           | ✅   | `decision_engine.py:_is_ambiguous_prompt()` + `_AMBIGUOUS_PROMPT_HINTS`             |
| 输出 Task Pattern      | ✅   | `classify()` 返回 `pattern` + `pattern_confidence`                                  |
| 输出初始 Complexity    | ✅   | `classify()` 返回 `complexity_score` + `complexity_label` + `complexity_confidence` |
| 输出初始模型建议       | ✅   | `decide()` 将 complexity_label 映射到 `final_model`                                 |
| 轻量设计（非重型推理） | ✅   | 默认模型 `deepseek-v4-flash`，max_tokens=512，temperature=0                         |

**注**: 设计文档要求"不需要是重型 LLM 推理器"，实现选用 deepseek-v4-flash（轻量）符合设计意图。

## 4.2 Runtime Complexity Tracker

**设计要求**: 基于 tool call 权重体系持续累积 Runtime Complexity Score。信号包括 tool 名称/参数/结果、文件路径、文件行数、grep/search 规模、bash exit code、错误/失败/重复编辑。

**实现状态**: ✅ 基本对齐

| 检查点                | 状态          | 证据                                                                                             |
| --------------------- | ------------- | ------------------------------------------------------------------------------------------------ |
| tool 名称权重         | ✅            | `runtime_score.py:_compute_delta()` — `tool_w` 维度                                              |
| 文件路径/类型权重     | ✅            | `runtime_score.py:_compute_delta()` — `file_type` 维度                                           |
| 文件行数权重          | ⚠️ 预留未激活 | `runtime_tracker.py:86` 注释 "Stage 7 引入"；`runtime_score.py` 接口已支持 `file_lines` 参数     |
| runtime_signal 权重   | ⚠️ 预留未激活 | `runtime_tracker.py:87` 注释 "Stage 5 引入"；`runtime_score.py` 接口已支持 `runtime_signal` 参数 |
| PostToolUse hook 接入 | ✅            | `post_tool_handler.py:dispatch()` 每次工具调用都走 `_tracker.track()`                            |
| Score 持久化          | ✅            | `runtime_tracker.py:_save()` 写入 `model_router_state_{sid}.json`                                |
| 只增不减              | ✅            | `RuntimeScore.accumulate()` 只做加法，无减分逻辑                                                 |

**差异**: 信号维度预留了 `file_lines` 和 `runtime_signal` 两个字段，但 `_convert()` 方法尚未实际提取这些信号。设计文档 §7.1 列出的信号源（bash exit code、grep/search 规模、测试失败等）仅部分覆盖。

## 4.3 TodoWrite Analyzer

**设计要求**: 首次 TodoWrite 触发 LLM 复杂度分析，识别任务规模、跨文件修改、测试/迁移/回归验证等。伪代码显示 `is_first_todo_write` 判断 + `llm_analyze_todos()` + `force_upgrade()`。

**实现状态**: ⚠️ 简化实现

| 检查点              | 状态        | 证据                                                                    |
| ------------------- | ----------- | ----------------------------------------------------------------------- |
| TodoWrite 检测      | ✅          | `post_tool_handler.py:65` `if tool_name == "TodoWrite"`                 |
| 任务分析            | ✅          | `todowrite_analyzer.py:analyze()` — 分析 todos 内容                     |
| 实施信号检测        | ✅          | `todowrite_analyzer.py:_is_implementation()` — 关键词匹配               |
| 首次 TodoWrite 判断 | ❌ 未实现   | 设计文档伪代码明确要求 `is_first_todo_write`，实现未区分首次/后续       |
| LLM 深度分析        | ❌ 未实现   | 设计文档要求 `llm_analyze_todos(todos)`，实现使用纯关键词匹配（非 LLM） |
| 跨文件修改检测      | ❌ 未实现   | 设计文档 §9.2 要求检测"是否跨文件/跨模块"，实现未做                     |
| 依赖关系分析        | ❌ 未实现   | 设计文档 §9.2 要求检测 todo 依赖关系，实现未做                          |
| force_upgrade 机制  | ✅ 间接实现 | `maybe_redecide()` 中 `todo_force_lock=True` 触发升级+锁定              |

**差异**: 这是目前发现的最显著差异。设计文档将 TodoWrite Analyzer 定位为"最关键的早期强信号"，要求使用 LLM 深度分析。但实现降级为纯关键词匹配，缺少：

1. `is_first_todo_write` 首次判断
2. LLM 深度分析（非关键词）
3. 跨文件/跨模块检测
4. 依赖关系分析
5. 与 Runtime Score 一致性校验

**严重程度**: 中 — TodoWrite 分析是路由决策的关键环节，简化可能降低复杂任务识别准确率。

## 4.4 Model Router

**设计要求**: 综合 Prompt 先验、Runtime Score、TodoWrite 分析结果，做最终路由决定。默认低配启动，按复杂度路由到低/中/高配模型。

**实现状态**: ✅ 对齐

| 检查点       | 状态 | 证据                                                                      |
| ------------ | ---- | ------------------------------------------------------------------------- |
| 综合三路信号 | ✅   | `maybe_redecide()` 融合 `current_label` + `runtime_label` + `todo_label`  |
| 默认低配启动 | ✅   | `decide()` 首次返回 `locked=False`，`final_model` 基于 prompt 分类        |
| 三档模型映射 | ✅   | `_COMPLEXITY_TO_MODEL`: simple/medium→MiniMax-M3, complex→deepseek-v4-pro |
| 最终路由决定 | ✅   | `maybe_redecide()` 升级+锁定 = 终裁                                       |

---

## 4.5 Decision Lock / Session State

**设计要求**: 路由结果写入 session state，后续请求不再重新计算，不允许降级。下次新 prompt 才重新开始。

**实现状态**: ✅ 对齐

| 检查点               | 状态 | 证据                                                      |
| -------------------- | ---- | --------------------------------------------------------- |
| 锁定机制             | ✅   | `DecisionRecord.locked` + `maybe_redecide()` 的 lock 检查 |
| Session state 持久化 | ✅   | `state_persistence.py:SessionStateStore` 单文件读写       |
| 不降级               | ✅   | `_max_complexity()` 只取较高者                            |
| Per-prompt 隔离      | ✅   | `decide()` 每次新 `prompt_id`，不跨 session 继承          |
| 状态机约束           | ✅   | `session_state_machine.py` 7 态不可逆转移                 |

## 总体评估

| 模块                              | 对齐度                                                |
| --------------------------------- | ----------------------------------------------------- |
| 4.1 Prompt Complexity Classifier  | ✅ 100%                                               |
| 4.2 Runtime Complexity Tracker    | ✅ 85% — file_lines/runtime_signal 预留未激活         |
| 4.3 TodoWrite Analyzer            | ⚠️ 55% — LLM 分析降级为关键词，缺少首次判断和深度分析 |
| 4.4 Model Router                  | ✅ 100%                                               |
| 4.5 Decision Lock / Session State | ✅ 100%                                               |

**综合评分**: 88% 对齐

**关键差异**:

1. **TodoWrite Analyzer 简化幅度最大**：设计文档要求的 LLM 深度分析降级为纯关键词匹配，缺少首次判断、跨文件检测、依赖关系分析
2. Runtime Tracker 信号维度未完全激活（file_lines、runtime_signal）
3. 其余三个模块对齐度很高
