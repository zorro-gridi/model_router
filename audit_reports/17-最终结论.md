| **对 session 连贯性更友好** | ⚠️       | 锁定后不切换（✅），但 Context Summary 缺失导致模型升级时缺少上下文简报                  |

### 可调参差距

设计文档期望通过 YAML 配置文件调整权重。实际上：

- `config/decision_weights.yaml` 存在但 `runtime_score.py` 不读取
- `stage_config.load_yaml_weights()` 已实现但无调用方
- 修改权重需要改 `stage_config._PLACEHOLDER_WEIGHTS` 字典（Python 代码）

**影响**: 低。权重系统本身的 2/4 维度未激活，调参需求尚不紧迫。

### Session 连贯性差距

设计文档期望模型升级时注入上下文摘要。实际上：

- Claude Code 框架自动传递完整 message history（部分弥补）
- 但缺少结构化的"任务进度简报"（§11 详述）

**影响**: 低。完整 history 传递避免了模型"从零开始"，摘要注入属于锦上添花。

## 17.3 核心工程原则验证

**设计文档损失函数**:

> **"复杂任务发现晚的代价，远高于简单任务误升的代价。"**

**实现对齐验证**:

| 安全机制     | 实现                                              | 如何体现损失函数       |
| ------------ | ------------------------------------------------- | ---------------------- |
| 保守偏置     | `_apply_conservative_bias()` → ambiguous ≥ medium | 宁可高估不低估         |
| 低升级阈值   | score > 70 → complex（满分 100）                  | 快速升级，减少发现延迟 |
| only-upgrade | `_max_complexity()` 取较高者                      | 不因后续安静而降级     |
| 早期信号优先 | `maybe_redecide()` 每 tool 都检查                 | 尽早发现尽早升级       |
| 锁定后不降   | `locked=True` → `maybe_redecide()` 返回 None      | 决策稳定性             |

**这条损失函数贯穿了整个实现**。每一个核心设计决策（保守偏置、低阈值、only-upgrade、快速锁定）都直接服务于"宁可误升，不可迟发现"的原则。
