# 审计报告：第10章 — 模型路由策略

**审计日期**: 2026-06-15
**设计文档版本**: V1.3
**审计范围**: 路由目标、路由建议、保守偏置 vs 实际实现

---

## 10.1 路由目标

**设计要求**: 尽早识别复杂任务、尽量少让复杂任务在低配模型上耗损、保证简单任务成本可控。

**实现状态**: ✅ 对齐

| 目标             | 实现                                     | 证据                                        |
| ---------------- | ---------------------------------------- | ------------------------------------------- |
| 尽早识别复杂任务 | 三阶段递进：prompt → runtime → todowrite | `decide()` → `track()` → `maybe_redecide()` |
| 减少低配损耗     | 模糊任务强制 medium，不启动为 simple     | `_apply_conservative_bias()`                |
| 简单任务成本可控 | simple → MiniMax-M3（低成本模型）        | `_COMPLEXITY_TO_MODEL["simple"]`            |

---

## 10.2 路由建议

**设计要求**: 三档 — simple→低配, medium→中配, complex→高配。

**实现状态**: ✅ 对齐

| 复杂度  | 设计模型 | 实际映射        | 状态 |
| ------- | -------- | --------------- | ---- |
| simple  | 低配模型 | MiniMax-M3      | ✅   |
| medium  | 中配模型 | MiniMax-M3      | ✅   |
| complex | 高配模型 | deepseek-v4-pro | ✅   |

**注**: simple 和 medium 当前映射到同一模型（MiniMax-M3），符合设计文档"低配/中配"的定位——两者都是成本可控的选项，差异在升级门槛而非模型本身。如果后续引入第三档模型（如 deepseek-v4-flash 作为 low 档），只需修改 `_COMPLEXITY_TO_MODEL` 即可。

---

## 10.3 保守偏置

**设计要求**: 模糊情况宁可往高配靠、复杂信号连续出现就尽早升级、不要等到百分百确认才升级、升级后不再回退。

**实现状态**: ✅ 对齐

| 偏置规则                  | 实现                                           | 证据                            |
| ------------------------- | ---------------------------------------------- | ------------------------------- |
| 模糊 → 高配               | `_apply_conservative_bias()` 强制 medium       | `decision_engine.py:124-130`    |
| 连续复杂信号 → 升级       | RuntimeScore 持续累积，不归零                  | `runtime_score.py:accumulate()` |
| 不等待百分百确认          | 阈值偏低：>70 即 complex，非>90                | `_label_from_score()`           |
| 升级后不回退              | `_max_complexity()` + locked 检查              | `decision_engine.py:152-156`    |
| 实施类 TodoWrite → 立即锁 | `todo_force_lock=True` 不等 runtime_score 达标 | `maybe_redecide():218`          |
