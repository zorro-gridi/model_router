# 审计报告：第5章 — 任务分类体系

**审计日期**: 2026-06-15
**设计文档版本**: V1.3
**审计范围**: Task Pattern 和 Task Complexity 分类 vs 实际实现

---

## 5.1 Task Pattern

**设计要求**: 8 种任务模式 — `explore`, `implement`, `debug`, `refactor`, `test`, `migrate`, `docs`, `ops`。Task Pattern 辅助判断复杂度来源，不直接决定模型。

**实现状态**: ⚠️ 分类枚举不一致

| 设计文档 Pattern | LLM 分类器 Pattern | 匹配        |
| ---------------- | ------------------ | ----------- |
| `explore`        | （无，归入 stage） | ❌          |
| `implement`      | `feature`          | ❌ 命名不同 |
| `debug`          | `bugfix`           | ❌ 命名不同 |
| `refactor`       | `refactor`         | ✅          |
| `test`           | `test`             | ✅          |
| `migrate`        | `migration`        | ⚠️ 命名近似 |
| `docs`           | `docs`             | ✅          |
| `ops`            | （无对应）         | ❌ 缺失     |

**LLM 分类器实际枚举** (`llm_classifier.py:324-327`):

```
feature, bugfix, refactor, test, research, migration, architecture, docs, audit
```

| 检查点                                 | 状态    | 证据                                                                            |
| -------------------------------------- | ------- | ------------------------------------------------------------------------------- |
| 设计文档 8 种 pattern                  | —       | explore, implement, debug, refactor, test, migrate, docs, ops                   |
| 实现 9 种 pattern                      | ⚠️ 不同 | feature, bugfix, refactor, test, research, migration, architecture, docs, audit |
| `explore` 缺失                         | ❌      | 设计文档有，LLM 分类器无（归入 stage 维度）                                     |
| `ops` 缺失                             | ❌      | 设计文档有，LLM 分类器无                                                        |
| `research`/`architecture`/`audit` 多出 | ⚠️      | 设计文档无，LLM 分类器有                                                        |
| `implement`→`feature` 命名差异         | ⚠️      | 语义相近但字符串不匹配                                                          |
| `debug`→`bugfix` 命名差异              | ⚠️      | 语义相近但字符串不匹配                                                          |
| Pattern 不直接决定模型                 | ✅      | `decide()` 中 pattern 仅记录，不影响 `final_model` 选择                         |

**差异分析**: 设计文档和 LLM 分类器的 pattern 枚举存在系统性命名差异。虽然语义覆盖大体一致（feature≈implement, bugfix≈debug, migration≈migrate），但 `ops` 完全缺失，`research`/`architecture`/`audit` 是额外的。Pattern 不直接决定路由（对齐设计），但枚举不一致意味着如果未来需要按 pattern 调整路由阈值，映射会出错。

## 5.2 Task Complexity

**设计要求**: 三档 — `simple`, `medium`, `complex`。不要再拆得过细。

**实现状态**: ✅ 对齐

| 检查点       | 状态 | 证据                                                              |
| ------------ | ---- | ----------------------------------------------------------------- |
| 三档分类     | ✅   | `VALID_COMPLEXITY_LABELS = {"simple", "medium", "complex"}`       |
| 分数阈值     | ✅   | 0-30 simple, 31-70 medium, 71-100 complex (`_label_from_score()`) |
| 不拆细       | ✅   | 仅三档，无额外子分类                                              |
| 用于最终路由 | ✅   | `_COMPLEXITY_TO_MODEL` 映射到模型                                 |

## 总体评估

| 分类维度            | 对齐度                              |
| ------------------- | ----------------------------------- |
| 5.1 Task Pattern    | ⚠️ 65% — 枚举命名不一致，`ops` 缺失 |
| 5.2 Task Complexity | ✅ 100%                             |

**综合评分**: 83% 对齐

**关键差异**:

1. Pattern 枚举存在命名差异：设计文档用 `implement`/`debug`，LLM 分类器用 `feature`/`bugfix`
2. 设计文档中的 `ops` 和 `explore` pattern 在 LLM 分类器中缺失
3. LLM 分类器多出 `research`/`architecture`/`audit` pattern
4. 这不影响路由正确性（pattern 不直接决定模型），但存在维护风险
