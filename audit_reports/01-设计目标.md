# 审计报告：第1章 — 设计目标

**审计日期**: 2026-06-15
**设计文档版本**: V1.3
**审计范围**: 4个核心目标 vs 实际实现

---

## 目标 1：放弃不可观测的 Stage 路由

**设计要求**: 不再尝试推断 Explore / Implement / Review 这类内部状态，Stage 不作为路由输入。

**实现状态**: ⚠️ 部分对齐

| 检查点                       | 状态    | 证据                                                                                                                 |
| ---------------------------- | ------- | -------------------------------------------------------------------------------------------------------------------- |
| 路由决策不使用 Stage         | ✅ 对齐 | `decision_engine.py` 的 `decide()` 和 `maybe_redecide()` 仅使用 `task_pattern` 和 `task_complexity`，不含 stage 字段 |
| DecisionRecord 不含 stage    | ✅ 对齐 | `decision_engine.py:39-43` REQUIRED_FIELDS 不含 stage                                                                |
| LLM 分类器仍输出 stage       | ⚠️ 残留 | `llm_classifier.py:84-93` system prompt 要求 LLM 返回 stage 维度（9种 stage）                                        |
| stage 分类基础设施完整保留   | ⚠️ 残留 | `stage_detector.py`（51KB）、`stage_config.py` 的 `STAGE_CONFIG`、`stage_show.py`、`stage` CLI 均在运行              |
| proxy.py 依赖 stage_detector | ⚠️ 残留 | `proxy.py` 通过 `stage_detector` 管理 active_session 指针和 stage 状态文件                                           |

**差异分析**: V1.3 核心路由逻辑已移除 Stage 依赖，但 LLM 分类器仍然每次调用都返回 stage 维度（浪费 token），且 stage_detector/stage_config 基础设施未按 §16 推荐删除项执行清理。stage 分类目前作为"影子信息"存在，不参与路由但仍消耗资源。

**严重程度**: 低 — 不影响路由正确性，但存在技术债务。

---

## 目标 2：尽早完成复杂度定性

**设计要求**: 在冷启动和早期运行阶段尽快决定本次 session 使用低配还是高配模型。

**实现状态**: ✅ 对齐

| 检查点                         | 状态      | 证据                                                                 |
| ------------------------------ | --------- | -------------------------------------------------------------------- |
| Prompt 先验分类（冷启动）      | ✅ 已实现 | `llm_classifier.py:classify()` 在首次 prompt 时运行                  |
| Runtime Score 累积（早期运行） | ✅ 已实现 | `runtime_tracker.py` 通过 PostToolUse hook 持续累积                  |
| 首次 TodoWrite 强信号          | ✅ 已实现 | `todowrite_analyzer.py` + `post_tool_handler.py:_handle_todowrite()` |
| 三阶段递进决策                 | ✅ 已实现 | `decide()` → runtime 累积 → `maybe_redecide()`                       |

---

## 目标 3：偏保守路由

**设计要求**: 宁可稍早升级，也不要让复杂任务长期停留在低配模型里。

**实现状态**: ✅ 对齐

| 检查点                    | 状态      | 证据                                                                                     |
| ------------------------- | --------- | ---------------------------------------------------------------------------------------- |
| 模糊 prompt 强制 ≥ medium | ✅ 已实现 | `decision_engine.py:_apply_conservative_bias()` + `_AMBIGUOUS_PROMPT_HINTS`              |
| 只升不降                  | ✅ 已实现 | `decision_engine.py:_max_complexity()` 取较高者，注释写明 "only-upgrade-never-downgrade" |
| 决策源追踪                | ✅ 已实现 | `DecisionRecord.decision_source` 记录 "prompt" / "runtime" / "todowrite"                 |
| 升级阈值偏低              | ✅ 已实现 | score ≤30 → simple, ≤70 → medium, >70 → complex（门槛适中偏保守）                        |

---

## 目标 4：一次决策，整段锁定

**设计要求**: 路由结果一旦确定，在下一次 prompt 提交前不再变化。

**实现状态**: ✅ 对齐

| 检查点                     | 状态      | 证据                                                               |
| -------------------------- | --------- | ------------------------------------------------------------------ |
| DecisionRecord.locked 字段 | ✅ 已实现 | `decision_engine.py:DecisionRecord.locked`                         |
| 首次决策不锁定             | ✅ 已实现 | `decide()` 返回 `locked=False`，注释写明"首次决策可改"             |
| 锁定后拒绝重决策           | ✅ 已实现 | `maybe_redecide()` 首行检查 `decision.get("locked")` → return None |
| 状态机 LOCKED 终态         | ✅ 已实现 | `session_state_machine.py` LOCKED → COMPLETED，无降级路径          |
| 不跨 session 继承          | ✅ 已实现 | `decide()` 每次接收新的 `prompt_id`，不携带上轮 score              |

---

## 总体评估

| 目标                | 对齐度                            |
| ------------------- | --------------------------------- |
| 1. 放弃 Stage 路由  | ⚠️ 85% — 路由已对齐，基础设施残留 |
| 2. 尽早复杂度定性   | ✅ 95%                            |
| 3. 偏保守路由       | ✅ 100%                           |
| 4. 一次决策整段锁定 | ✅ 100%                           |

**综合评分**: 95% 对齐

**关键差异**:

1. Stage 分类基础设施未按 §16 推荐删除（不影响路由正确性）
2. 无功能缺失

**建议**:

- 考虑从 `llm_classifier.py` 的 system prompt 中移除 stage 维度，减少 token 消耗
- 评估是否可在后续版本中移除 `stage_detector.py` / `STAGE_CONFIG` 对 proxy.py 的耦合
