# 审计报告：第3章 — 设计原则

**审计日期**: 2026-06-15
**设计文档版本**: V1.3
**审计范围**: 4条设计原则 vs 实际实现

---

## 3.1 放弃 Stage 路由

**设计要求**: Stage 不作为路由输入，只作为内部执行现象参考。

**实现状态**: ⚠️ 部分对齐

| 检查点                                          | 状态 | 证据                                                        |
| ----------------------------------------------- | ---- | ----------------------------------------------------------- |
| V1.3 决策核心不使用 stage                       | ✅   | `decision_engine.py` 不含 stage import，不含 stage 字段     |
| LLM 分类器仍输出 stage                          | ⚠️   | `llm_classifier.py:84-93` 9 种 stage 枚举，每次调用都输出   |
| stage_config 的 STAGE_CONFIG 仍被 proxy.py 使用 | ⚠️   | proxy.py 用它做 model→base_url/api_key 映射，属于基础设施层 |
| stage 信息被写入 state_index.json               | ⚠️   | `proxy.py:1015` 仍调用 `stage_detector._update_state_index` |

**差异**: Stage 在路由中已"边缘化"但不彻底——仍在分类器 token 消耗中占一份，在 proxy 层作为模型元数据存在。设计文档要求"只作为内部执行现象参考"，当前实现与其基本一致（stage 仅用于日志/展示，不用于路由）。

---

## 3.2 只做早期决策，不做后期反复切换

**设计要求**: 复杂度判断在 session 早期完成，判断后立即锁定，不做降级。

**实现状态**: ✅ 对齐

| 检查点                  | 状态 | 证据                                                                |
| ----------------------- | ---- | ------------------------------------------------------------------- |
| 早期判断（prompt 级别） | ✅   | `decide()` 在 prompt 到来时立即运行                                 |
| 判断后锁定              | ✅   | `maybe_redecide()` 遇到 locked=True 直接返回 None                   |
| 不做降级                | ✅   | `_max_complexity()` 只取较高者；注释 "only-upgrade-never-downgrade" |
| 状态机无降级路径        | ✅   | `session_state_machine.py` 转移表无降级边                           |

---

## 3.3 保守优先，升级门槛偏低

**设计要求**: 模糊任务宁可略早升级；高风险任务直接高配；低复杂度才允许低配。

**实现状态**: ✅ 对齐

| 检查点                           | 状态 | 证据                                                                  |
| -------------------------------- | ---- | --------------------------------------------------------------------- |
| 模糊 prompt → 强制 ≥ medium      | ✅   | `_apply_conservative_bias()` + `_AMBIGUOUS_PROMPT_HINTS`              |
| 显式高复杂信号 → 偏 high         | ✅   | `_is_ambiguous_prompt()` 中 `has_explicit` 排除"迁移/架构/跨模块"等   |
| 复杂度阈值偏低（≤30 simple）     | ✅   | `_label_from_score()`: 0-30 simple, 31-70 medium, 71-100 complex      |
| 实施类 TodoWrite → 强制 ≥ medium | ✅   | `maybe_redecide()`: `is_implementation=True` → `todo_force_lock=True` |

---

## 3.4 权重支持自定义配置

**设计要求**: tool call / 文件类型 / 文件行数权重可外部自定义，推荐 YAML 格式。

**实现状态**: ⚠️ 部分实现

| 检查点            | 状态            | 证据                                                               |
| ----------------- | --------------- | ------------------------------------------------------------------ |
| 权重可注入        | ✅              | `RuntimeScore.__init__(weights=custom)` 支持覆盖                   |
| 默认权重存在      | ✅              | `stage_config._PLACEHOLDER_WEIGHTS` 提供默认值                     |
| YAML 格式配置文件 | ❌ 未实现       | 权重定义在 Python dict 中（`stage_config.py`），无独立 YAML 文件   |
| 文件类型权重      | ✅              | `runtime_score.py:_compute_delta()` 支持 `file_type` 权重维度      |
| 文件行数权重      | ⚠️ 预留但未接入 | `runtime_tracker.py:86` 注释 "file_lines 暂不计算（Stage 7 引入）" |
| 项目子目录权重    | ❌ 未实现       | 设计文档 §7.1 "按项目子目录分配权重" 未找到实现                    |

**差异**: 权重体系的核心机制已建立（`RuntimeScore` 支持三维度权重注入），但：

- 未按设计文档推荐使用 YAML 格式（仅 Python dict）
- 文件行数维度预留但未激活
- 项目子目录维度未实现

## 总体评估

| 原则                   | 对齐度                                    |
| ---------------------- | ----------------------------------------- |
| 3.1 放弃 Stage 路由    | ⚠️ 85%                                    |
| 3.2 早期决策不反复切换 | ✅ 100%                                   |
| 3.3 保守优先           | ✅ 100%                                   |
| 3.4 权重自定义配置     | ⚠️ 70% — 机制存在，YAML/行数/子目录未落地 |

**综合评分**: 89% 对齐

**关键差异**:

1. 权重配置未使用 YAML 格式（设计文档 §7.1 推荐项）
2. 文件行数和项目子目录权重维度预留但未实现
3. Stage 未彻底移除（同第1/2章结论）
