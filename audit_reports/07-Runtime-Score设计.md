# 审计报告：第7章 — Runtime Complexity Score 设计

**审计日期**: 2026-06-15
**设计文档版本**: V1.3
**审计范围**: 评分原则、权重规则、文件权重、降级禁止 vs 实际实现

## 7.1 评分原则

**设计要求**:

- 读核心文件 > 读说明文件
- 大范围 grep/search > 单次读取
- 测试失败 > 普通执行
- 编辑核心逻辑文件 > 编辑配置文件
- 重复 read/edit/test 循环 → 复杂度上升
- 权重配置文件推荐 YAML 格式
- 按项目子目录分配权重

**实现状态**: ⚠️ 部分对齐

| 检查点         | 状态                | 证据                                                                                                                                                                    |
| -------------- | ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 工具类型权重   | ✅                  | `_PLACEHOLDER_WEIGHTS["tool"]` 13 种工具，TodoWrite=8 最高                                                                                                              |
| 文件类型权重   | ✅                  | `_PLACEHOLDER_WEIGHTS["file_type"]` 按扩展名分层：`.py`=3, `.md`=1, `.lock`=0                                                                                           |
| 文件行数权重   | ⚠️ 定义但未激活     | `file_lines` 权重定义了 small=1/medium=2/large=3，但 `runtime_tracker._convert()` 不提取此信号                                                                          |
| 运行时信号权重 | ⚠️ 定义但未激活     | `runtime_signal` 权重定义了 bash_nonzero_exit=4/test_failure=5 等，但 `_convert()` 不提取                                                                               |
| YAML 配置      | ❌ 文件存在但未接通 | `config/decision_weights.yaml` 已编写，`stage_config.get_weights()` 已实现加载器，但 `runtime_score.py:28-30` 硬编码导入 `_PLACEHOLDER_WEIGHTS`，未调用 `get_weights()` |
| 项目子目录权重 | ❌ 未实现           | 设计文档 §7.1 要求按子目录分配权重，无对应实现                                                                                                                          |
| 重复编辑检测   | ❌ 未实现           | `runtime_signal.edit_retry_loop` 权重已定义但无检测逻辑                                                                                                                 |

**严重差异**: YAML 配置文件和加载器已完整实现，但 `runtime_score.py` 硬编码使用 `_PLACEHOLDER_WEIGHTS`，导致 YAML 配置实际不生效。这是**断路**问题——基础设施完整但连接断开。

## 7.2 Tool Score 基本规则

**设计要求**: 内置固定权重 + 用户可自定义。示例：Read README/docs 低权重、读核心文件 中高权重、Grep/Search 大范围命中 高权重、Bash pytest/test 中高权重、Bash 非零退出码 高权重、Edit 核心业务文件 高权重、TodoWrite 首次出现 极高权重、重复失败+重试 持续加权。

**实现状态**: ✅ 权重定义对齐，⚠️ 规则覆盖不完整

| 设计要求                      | 实现                                                    | 状态 |
| ----------------------------- | ------------------------------------------------------- | ---- |
| Read README/docs 低权重       | `.md`=1, `.txt`=1 文件类型权重                          | ✅   |
| Read 核心文件 中高权重        | `.py`=3, `.go`=3 等源码文件                             | ✅   |
| Grep/Search 大范围命中 高权重 | `grep_large_hits`=3, `search_large_hits`=3 定义但未激活 | ⚠️   |
| Bash test 中高权重            | 无特殊 test 检测（Bash=2 统一）                         | ⚠️   |
| Bash 非零退出码 高权重        | `bash_nonzero_exit`=4 定义但未激活                      | ⚠️   |
| Edit 核心文件 高权重          | `Edit`=4 + 文件类型叠加                                 | ✅   |
| TodoWrite 首次 极高权重       | `TodoWrite`=8 最高；但未区分首次/后续                   | ⚠️   |
| 重复失败+重试 持续加权        | `edit_retry_loop`=2 定义但未激活                        | ⚠️   |

## 7.3 文件权重规则

**设计要求**: 文件类型分层（核心业务/模型训练 高、控制器/编排 中高、测试 中、配置 低到中、文档 低、生成文件 极低）+ 行数分层 + 组合规则。

**实现状态**: ⚠️ 文件类型部分对齐，行数/组合未激活

| 检查点                    | 状态 | 证据                                                                                            |
| ------------------------- | ---- | ----------------------------------------------------------------------------------------------- |
| 文件类型分层              | ✅   | 源码(.py,.go,.ts等)=3，配置(.yaml,.toml)=2，数据(.json)=1，文档(.md,.txt)=1，生成(.lock,.sum)=0 |
| 核心/测试/配置区分        | ⚠️   | 所有源码文件统一权重3，未区分核心业务文件 vs 测试文件                                           |
| 行数分层                  | ❌   | `file_lines` 权重定义完整但 `_convert()` 未计算文件行数                                         |
| 组合规则（读+改同一文件） | ❌   | 无跨事件关联逻辑                                                                                |
| 多文件联动修改            | ❌   | 无检测                                                                                          |

---

## 7.4 Runtime Score 不做降级判定

**设计要求**: Score 同一 session 只能增长不归零。用途是触发升级、确认复杂任务、兜底识别。

**实现状态**: ✅ 对齐

| 检查点       | 状态 | 证据                                                    |
| ------------ | ---- | ------------------------------------------------------- |
| 只增不减     | ✅   | `RuntimeScore.accumulate()` 只做 `self._score += delta` |
| 不归零       | ✅   | 无 reset/clear 接口                                     |
| 用于升级触发 | ✅   | `maybe_redecide()` 读取 runtime_score 判断升级          |
| 兜底识别     | ✅   | 无 TodoWrite 时依赖 runtime_score 做决策                |
| 不做降级     | ✅   | `_max_complexity()` 只取较高者                          |
