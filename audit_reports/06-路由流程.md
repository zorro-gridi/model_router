# 审计报告：第6章 — 路由流程

**审计日期**: 2026-06-15
**设计文档版本**: V1.3
**审计范围**: 4阶段路由流程 vs 实际实现

---

## 6.1 冷启动阶段：Prompt 先验判断

**设计要求**: 用户首次提交 prompt 时做轻量分类，输出 Task Pattern、初始 Complexity、初始模型建议、置信度。规则：明确复杂→高配，明确简单→低配，模糊→低/中配 + 低升级门槛。

**实现状态**: ✅ 对齐

| 检查点              | 状态 | 证据                                                                  |
| ------------------- | ---- | --------------------------------------------------------------------- |
| 轻量分类            | ✅   | `llm_classifier.classify(prompt)` — deepseek-v4-flash, max_tokens=512 |
| 输出 Task Pattern   | ✅   | `classify()` 返回 `pattern` 字段                                      |
| 输出初始 Complexity | ✅   | `classify()` 返回 `complexity_label` + `complexity_score`             |
| 输出置信度          | ✅   | `classify()` 返回 `pattern_confidence` + `complexity_confidence`      |
| 明确复杂→高配       | ✅   | `_COMPLEXITY_TO_MODEL["complex"] = "deepseek-v4-pro"`                 |
| 明确简单→低配       | ✅   | `_COMPLEXITY_TO_MODEL["simple"] = "MiniMax-M3"`                       |
| 模糊→保守偏置       | ✅   | `_apply_conservative_bias()` 强制 medium                              |
| 初始模型建议        | ✅   | `decide()` 返回 `DecisionRecord.final_model`                          |

---

## 6.2 早期运行阶段：Runtime Score 累积

**设计要求**: Session 启动后通过 tool call 持续累积 Runtime Complexity Score。关注工具类型、输出规模、失败、大范围搜索、核心业务文件、重复修改和反复验证。

**实现状态**: ⚠️ 信号覆盖不完整

| 检查点                     | 状态      | 证据                                                 |
| -------------------------- | --------- | ---------------------------------------------------- |
| 工具类型权重               | ✅        | `runtime_score.py` tool 维度                         |
| 工具输出规模               | ❌ 未实现 | `_convert()` 未提取 tool_result 规模信息             |
| 是否失败（bash exit code） | ❌ 未实现 | `runtime_signal` 预留但 `_convert()` 未提取          |
| 大范围搜索命中             | ❌ 未实现 | Grep/Search 返回规模未检测                           |
| 核心业务文件               | ⚠️ 部分   | `file_type` 权重可区分，但"核心业务文件"依赖权重配置 |
| 重复修改/反复验证          | ❌ 未实现 | 无重复编辑检测逻辑                                   |
| Score 累积 + 持久化        | ✅        | `RuntimeTracker.track()` → accumulate → save         |

**差异**: 设计文档 §6.2 列出的 6 个信号源中，仅"工具类型"完全实现。"输出规模"、"失败检测"、"搜索规模"、"重复修改"均未接入。

---

## 6.3 第一次 TodoWrite 阶段：最终复杂度确认

**设计要求**: 首次 TodoWrite 触发 LLM 高权重分析。分析内容：todo 数量、依赖关系、跨文件修改、测试/迁移/回归验证、任务拆分复杂度、与 Runtime Score 一致性。

**实现状态**: ❌ 降级为关键词匹配

| 检查点                      | 状态 | 证据                                                   |
| --------------------------- | ---- | ------------------------------------------------------ |
| TodoWrite 触发              | ✅   | `post_tool_handler.py` 检测 `tool_name == "TodoWrite"` |
| todo 数量分析               | ✅   | `todowrite_analyzer.py` 统计 total/pending/completed   |
| 依赖关系分析                | ❌   | 纯文本解析未做依赖分析                                 |
| 跨文件修改检测              | ❌   | 未检测                                                 |
| 测试/迁移/回归验证          | ❌   | 关键词匹配不支持语义理解                               |
| 与 Runtime Score 一致性校验 | ❌   | `maybe_redecide()` 融合时不校验一致性                  |
| LLM 深度分析                | ❌   | 设计文档要求 `llm_analyze_todos()`，实现用关键词       |
| 首次判断                    | ❌   | 未区分首次/后续 TodoWrite                              |

**差异**: 这是路由流程中最关键的差异。设计文档将 TodoWrite 定位为"最关键的早期强信号"并明确要求 LLM 分析，但实现将其降级为纯关键词匹配，削弱了该信号在复杂任务识别中的价值。

---

## 6.4 决策锁定

**设计要求**: 路由结果写入 session state，当前 prompt 模型固定，后续不切换，不降级。本次 prompt 只经历一次路由收敛。

**实现状态**: ✅ 对齐

| 检查点             | 状态 | 证据                                                         |
| ------------------ | ---- | ------------------------------------------------------------ |
| 写入 session state | ✅   | `state_persistence.py:SessionStateStore.write()`             |
| 模型固定           | ✅   | `DecisionRecord.locked=True` 后 `maybe_redecide()` 返回 None |
| 不切换/不降级      | ✅   | `_max_complexity()` + locked 检查                            |
| 一次收敛           | ✅   | `decide()` → `maybe_redecide()` → lock，仅一次升级窗口       |

## 总体评估

| 阶段                   | 对齐度                                          |
| ---------------------- | ----------------------------------------------- |
| 6.1 Prompt 先验判断    | ✅ 100%                                         |
| 6.2 Runtime Score 累积 | ⚠️ 60% — 5/6 信号源未接入                       |
| 6.3 TodoWrite 最终确认 | ❌ 40% — LLM 分析降级为关键词，5/6 分析维度缺失 |

**综合评分**: 75% 对齐

**关键差异**:

1. **Runtime 信号覆盖严重不足**：仅工具类型生效，输出规模/失败/搜索规模/重复编辑均未实现
2. **TodoWrite 分析与设计差距最大**：LLM 分析→关键词匹配，6个分析维度仅覆盖1个（todo数量）
3. Prompt 先验和决策锁定对齐度最高
