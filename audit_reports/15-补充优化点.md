# 审计报告：第15章 — 补充的优化点

**审计日期**: 2026-06-15
**设计文档版本**: V1.3
**审计范围**: 5个补充优化点 vs 实际实现

---

## 15.1 引入"首次强信号优先"原则

**设计要求**: 多个信号同时出现时，优先使用最早的高置信度强信号，不等所有信号收集完。

**实现状态**: ✅ 对齐

| 检查点         | 状态 | 证据                                                                                  |
| -------------- | ---- | ------------------------------------------------------------------------------------- |
| 首个强信号优先 | ✅   | `maybe_redecide()` 每次 PostToolUse 都检查：runtime_score 达标或 TodoWrite 触发即升级 |
| 不等待所有信号 | ✅   | 不要求 runtime_score + todowrite 同时满足才升级                                       |
| 独立触发路径   | ✅   | runtime_score 和 todowrite_signal 各自独立触发升级                                    |

---

## 15.2 给复杂任务设置更低的升级阈值

**设计要求**: 不追求高准确率后再升级，追求第 1 次明显复杂信号出现时就升级。

**实现状态**: ✅ 对齐

| 检查点                | 状态 | 证据                                                 |
| --------------------- | ---- | ---------------------------------------------------- |
| 低升级阈值            | ✅   | score > 70 → complex（满分 100，70 即触发）          |
| 模糊 prompt 门槛更低  | ✅   | medium 起步（31 分），距 complex 仅 39 分            |
| 第 1 次复杂信号即升级 | ✅   | `promoted=True` 时立即写 `locked=True`，不等二次确认 |
| 不追求高准确率        | ✅   | 无二次验证/确认步骤                                  |

---

## 15.3 高风险任务直通高配

**设计要求**: 迁移、重构、支付、测试修复、跨模块改造等高风险任务设置更激进的直通规则。

**实现状态**: ⚠️ 部分实现

| 检查点             | 状态      | 证据                                                                                           |
| ------------------ | --------- | ---------------------------------------------------------------------------------------------- |
| 关键词提示高风险   | ✅        | `_is_ambiguous_prompt()` 中 `has_explicit` 包含 "迁移/架构/跨模块/测试修复/重构整个/从 0 到 1" |
| LLM 分类器语义识别 | ✅        | system prompt 中 complexity 71-100 对应 "cross-module, architectural, high risk"               |
| 显式直通规则       | ❌ 未实现 | 无类似 `if "迁移" in prompt: force complex` 的硬编码直通规则                                   |
| 低门槛升级         | ✅        | 即使 prompt 判断为 medium，runtime_score 也可快速触发 complex                                  |

**差异**: 设计文档建议"更激进的直通规则"，实现选择了依赖 LLM 语义判断 + 低升级门槛的组合，而非硬编码关键词直通。这是保守程度的选择差异，非功能缺失。

---

## 15.4 记录路由理由

**设计要求**: 每次 final route 记录一句简短理由（如 "prompt 显式高复杂"、"runtime score 快速上升"、"TodoWrite 表现出多步骤依赖"、"测试失败后升级"）。

**实现状态**: ⚠️ 部分实现

| 检查点               | 状态              | 证据                                                                       |
| -------------------- | ----------------- | -------------------------------------------------------------------------- |
| decision_source 字段 | ✅                | `"prompt"` / `"runtime"` / `"todowrite"`                                   |
| LLM 分类器 reasoning | ⚠️ 仅 prompt 阶段 | `llm_classifier.py` 返回 `reasoning` 字段，但 `DecisionRecord` 不保存      |
| runtime 升级理由     | ❌                | `decision_source="runtime"` 仅表明来源，不记录具体信号（哪个 tool/多少分） |
| todowrite 升级理由   | ❌                | `decision_source="todowrite"` 仅表明来源，不记录分析细节                   |
| 简短人类可读理由     | ❌                | 无类似 `"reason": "runtime_score 快速上升至 75"` 的字段                    |

**差异**: `decision_source` 提供了粗粒度的路由理由，但缺少设计文档示例中的细粒度人类可读理由（如 "runtime score 快速上升"、"测试失败后升级"）。

---

## 15.5 只在 session 级别锁定，不做跨 session 继承

**设计要求**: 下一次 prompt 重新判断，不把上一次 session 的 score 带入新任务，避免误污染。

**实现状态**: ✅ 对齐

| 检查点             | 状态 | 证据                                                           |
| ------------------ | ---- | -------------------------------------------------------------- |
| 新 prompt 重新判断 | ✅   | `decide()` 接收新 `prompt_id`，`runtime_score=0, todo_score=0` |
| 不继承上轮 score   | ✅   | `decide()` 中 `runtime_score=0` 硬编码，不读取旧 session state |
| Session state 隔离 | ✅   | `model_router_state_{sid}.json` 按 sid 隔离                    |

---

## 总体评估

| 优化点                 | 对齐度                                     |
| ---------------------- | ------------------------------------------ |
| 15.1 首次强信号优先    | ✅ 100%                                    |
| 15.2 更低升级阈值      | ✅ 100%                                    |
| 15.3 高风险直通高配    | ⚠️ 80% — 无显式直通规则，依赖 LLM + 低阈值 |
| 15.4 记录路由理由      | ⚠️ 60% — 有来源但无细粒度理由              |
| 15.5 不跨 session 继承 | ✅ 100%                                    |

**综合评分**: 88% 对齐

**关键差异**:

1. 路由理由缺少细粒度人类可读描述（仅 `decision_source` 粗粒度标签）
2. 高风险直通规则未硬编码，依赖 LLM 语义判断（选择差异，非缺失）
