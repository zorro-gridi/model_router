# 审计报告：第8章 — Prompt 先验分类策略

**审计日期**: 2026-06-15
**设计文档版本**: V1.3
**审计范围**: 三种 prompt 类型的分类策略 vs 实际实现

---

## 8.1 明确高复杂度请求

**设计要求**: 迁移整个支付系统、多模块联动重构、大范围测试修复、跨文件架构改造、新功能从0到1全链路实现 → 直接偏向高配。

**实现状态**: ✅ 对齐

| 检查点               | 状态 | 证据                                                                                           |
| -------------------- | ---- | ---------------------------------------------------------------------------------------------- |
| LLM 分类器识别高复杂 | ✅   | system prompt 中 complexity_score 71-100 = complex                                             |
| 关键词辅助识别       | ✅   | `_is_ambiguous_prompt()` 中 `has_explicit` 排除 "迁移/架构/跨模块/测试修复/重构整个/从 0 到 1" |
| 高复杂 → 高配模型    | ✅   | `_COMPLEXITY_TO_MODEL["complex"] = "deepseek-v4-pro"`                                          |
| 直通高配             | ✅   | `decide()` 中 complexity_label="complex" → final_model="deepseek-v4-pro"                       |

**注**: 设计文档列出的高风险场景（迁移、重构、测试修复等）在 `_is_ambiguous_prompt()` 中被识别为"显式信号"（不触发模糊偏置），LLM 分类器进一步做语义级判断。

---

## 8.2 明确低复杂度请求

**设计要求**: 小范围修 bug、单文件文案修改、简单脚本修正、局部格式整理 → 先低配。

**实现状态**: ✅ 对齐

| 检查点               | 状态 | 证据                                                         |
| -------------------- | ---- | ------------------------------------------------------------ |
| LLM 分类器识别低复杂 | ✅   | system prompt 中 complexity_score 0-30 = simple              |
| 低复杂 → 低配模型    | ✅   | `_COMPLEXITY_TO_MODEL["simple"] = "MiniMax-M3"`              |
| 仍可升级             | ✅   | `decide()` 返回 `locked=False`，Runtime/TodoWrite 可触发升级 |

---

## 8.3 模糊请求

**设计要求**: "帮我修一下这个问题"、"看下为什么报错"、"帮我优化一下逻辑" → 不依赖 prompt 做强判断，让 Runtime 和 TodoWrite 尽快接管。

**实现状态**: ✅ 对齐

| 检查点             | 状态 | 证据                                                                                   |
| ------------------ | ---- | -------------------------------------------------------------------------------------- |
| 模糊 prompt 检测   | ✅   | `_AMBIGUOUS_PROMPT_HINTS`: "帮我", "看下", "怎么", "优化", "重构", "改进", "调一下" 等 |
| 模糊 → 强制 medium | ✅   | `_apply_conservative_bias()` 将 simple→medium                                          |
| 低升级门槛         | ✅   | medium 起始（31 分），只需 40 分增量即可触发 complex（70 阈值）                        |
| 首次决策不锁定     | ✅   | `decide()` locked=False，为 Runtime/TodoWrite 留升级空间                               |
| LLM 语义判断兜底   | ✅   | 即使关键词判定模糊，LLM 分类器仍做语义级评分                                           |

**差异**: 设计文档说模糊任务"默认低配或中配"，实现选择了更保守的策略——强制 medium（不低于中配）。这符合 §10.3 "保守偏置"原则，属于**合理偏离**。

## 整体分类流程验证

以三个典型 prompt 验证路由：

| Prompt 示例                | LLM 分类    | 保守偏置                             | 初始模型        | locked |
| -------------------------- | ----------- | ------------------------------------ | --------------- | ------ |
| "迁移整个支付系统到新架构" | complex(85) | 无影响（已是 complex）               | deepseek-v4-pro | false  |
| "修一下 README 里的 typo"  | simple(10)  | 无影响（非模糊）                     | MiniMax-M3      | false  |
| "帮我优化一下这个逻辑"     | medium(45)  | 关键词匹配"帮我"+"优化"→ 至少 medium | MiniMax-M3      | false  |

全部符合设计预期。
