## 转移路径对比

**设计文档 §12 定义的链路**:

```
Idle → PromptReceived → Preclassified → RuntimeAccumulating → TodoTriggered → FinalRouted → Locked
```

**实现 `session_state_machine.py` 定义的链路**:

```
INITIAL → PROMPT_SEEN → PROMPT_PREDICTED → RUNTIME_ACCUMULATING → TODOWRITE_SIGNAL → LOCKED → COMPLETED
                                                                     ↘ LOCKED (threshold_reached)
```

## 逐状态对比

### 12.1 初始态 (Idle → PromptReceived)

**设计**: `Idle → PromptReceived`
**实现**: `INITIAL + "prompt_received" → PROMPT_SEEN`
**状态**: ✅ 对齐

### 12.2 先验判定 (PromptReceived → Preclassified)

**设计**: `PromptReceived → Preclassified`
**实现**: `PROMPT_SEEN + "decision_made" → PROMPT_PREDICTED`
**状态**: ✅ 对齐

### 12.3 运行累积 (Preclassified → RuntimeAccumulating)

**设计**: `Preclassified → RuntimeAccumulating`
**实现**: `PROMPT_PREDICTED + "tool_started" → RUNTIME_ACCUMULATING`
**状态**: ✅ 对齐

### 12.4 黄金信号触发 (RuntimeAccumulating → TodoTriggered)

**设计**: `RuntimeAccumulating → TodoTriggered`
**实现**: `RUNTIME_ACCUMULATING + "todowrite_detected" → TODOWRITE_SIGNAL`
**状态**: ✅ 对齐

### 12.5 最终决策 (TodoTriggered → FinalRouted)

**设计**: 有独立的 `FinalRouted` 状态
**实现**: 无独立 `FinalRouted` 状态；直接通过转移边到达 LOCKED：

- `RUNTIME_ACCUMULATING + "threshold_reached" → LOCKED`
- `TODOWRITE_SIGNAL + "threshold_reached" → LOCKED`
  **差异**: 实现合并了 FinalRouted 和 Locked 的转移，但语义等价——两个路径最终都到达 LOCKED

### 12.6 锁定 (FinalRouted → Locked)

**设计**: `FinalRouted → Locked`
**实现**: LOCKED 之后还可转移到 COMPLETED（`LOCKED + "session_ended" → COMPLETED`）
**差异**: 实现多一个 COMPLETED 终态。设计文档未定义 session 结束后的状态

## DecisionLock 与状态机的集成

`decision_lock.py` 内嵌了状态机调用（`transition()` 方法委托给 `session_state_machine.transition()`），同时维护独立的 `_records` dict（决策锁定原语）和 `_states` dict（状态机状态）。两者在同一个 `DecisionLock` 实例中共存但职责分离：

- `try_acquire()` / `is_locked()` / `get()` → 决策锁
- `get_state()` / `transition()` → 状态机

这种设计使状态机作为决策锁的校验层存在，符合设计文档"锁定后直到下一次 prompt 才重新开始"的约束。
