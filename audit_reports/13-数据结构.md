## 13.1 Session State

**设计要求**:

```json
{
  "session_id": "xxx",
  "prompt_id": "xxx",
  "task_pattern": "implement",
  "task_complexity": "medium",
  "prompt_confidence": 0.72,
  "runtime_score": 38,
  "todo_score": 44,
  "final_model": "sonnet",
  "locked": true,
  "decision_source": "todo_write",
  "last_update": 1710000000
}
```

**实现**: `DecisionRecord` dataclass (`decision_engine.py:70-96`)

| 设计字段            | 实现字段                   | 匹配 |
| ------------------- | -------------------------- | ---- |
| `session_id`        | `session_id: str`          | ✅   |
| `prompt_id`         | `prompt_id: str`           | ✅   |
| `task_pattern`      | `task_pattern: str`        | ✅   |
| `task_complexity`   | `task_complexity: str`     | ✅   |
| `prompt_confidence` | `prompt_confidence: float` | ✅   |
| `runtime_score`     | `runtime_score: int`       | ✅   |
| `todo_score`        | `todo_score: int`          | ✅   |
| `final_model`       | `final_model: str`         | ✅   |
| `locked`            | `locked: bool`             | ✅   |
| `decision_source`   | `decision_source: str`     | ✅   |
| `last_update`       | `last_update: int`         | ✅   |

**11 个字段全部对齐，0 缺失，0 多余。** ✅

**持久化格式**: `DecisionRecord` 序列化后作为 `model_router_state_{sid}.json` 的 `decision` 子对象存储，顶层还包含 `version`、`session_id`、`last_update` 等管理字段。额外字段（`stage`、`pattern`、`complexity`、`batch`、`fallback`、`reqcnt`、`workflow_step`）作为可选扩展存在，不破坏核心 schema。

## 13.2 Runtime Event

**设计要求**:

```json
{
  "tool_name": "Read",
  "file_path": "src/payment/callback.py",
  "file_lines": 420,
  "signal_weight": 6,
  "timestamp": 1710000000
}
```

**实现**: `RuntimeScore._events` 列表中每个事件的实际格式 (`runtime_score.py:88`):

```python
{**event, "delta": delta}
# event = {"tool": str, "file_type": str, "file_lines": str, "runtime_signal": str}
```

| 设计字段                 | 实现字段                                     | 匹配                                  |
| ------------------------ | -------------------------------------------- | ------------------------------------- |
| `tool_name`              | `tool`                                       | ⚠️ 命名不同                           |
| `file_path`              | `file_type`（仅扩展名）                      | ⚠️ 设计要完整路径，实现只有扩展名     |
| `file_lines` (int: 420)  | `file_lines` (str: "small"/"medium"/"large") | ⚠️ 设计要具体行数，实现用分层标签     |
| `signal_weight` (int: 6) | `delta` (int)                                | ⚠️ 命名不同但语义等价（本次事件增量） |
| `timestamp`              | 无                                           | ❌ 缺失                               |

**差异分析**:

1. **file_path → file_type**: 设计文档期望记录完整文件路径（如 `src/payment/callback.py`），实现只保留扩展名（`.py`）。丢失了文件路径信息用于事后审计。
2. **file_lines 类型变化**: 设计文档期望具体行数（int），实现用分层标签（small/medium/large）。实现更简化，但丢失精度。
3. **timestamp 缺失**: 事件不记录时间戳，无法做时序分析。

## 13.3 Todo Analysis Result

**设计要求**:

```json
{
  "todo_count": 6,
  "cross_file": true,
  "has_tests": true,
  "has_migration": false,
  "todo_complexity": "complex",
  "confidence": 0.86
}
```

**实现**: `TodoWriteAnalyzer.analyze()` 返回值 (`todowrite_analyzer.py:52-58`):

```python
{
    "is_implementation": bool,
    "total": int,
    "pending": int,
    "completed": int,
    "complexity_signal": float,  # [0, 1]
}
```

| 设计字段                      | 实现字段                            | 匹配        |
| ----------------------------- | ----------------------------------- | ----------- |
| `todo_count`                  | `total`                             | ✅ 语义等价 |
| `cross_file`                  | 无                                  | ❌ 缺失     |
| `has_tests`                   | 无                                  | ❌ 缺失     |
| `has_migration`               | 无                                  | ❌ 缺失     |
| `todo_complexity` ("complex") | `complexity_signal` (0.0-1.0 float) | ⚠️ 格式不同 |
| `confidence`                  | 无                                  | ❌ 缺失     |
| （无）                        | `is_implementation`                 | ⚠️ 实现多出 |
| （无）                        | `pending`                           | ⚠️ 实现多出 |
| （无）                        | `completed`                         | ⚠️ 实现多出 |

**差异分析**: Todo Analysis Result 是 schema 差异最大的结构。设计文档的 6 个字段中仅 `todo_count` 有直接对应。`cross_file`、`has_tests`、`has_migration`、`confidence` 全部缺失——这与 §9 中 TodoWrite Analyzer 从 LLM 分析降级为关键词匹配直接相关（关键词匹配无法判断跨文件、测试、迁移等语义信息）。

| 数据结构                  | 对齐度                                    |
| ------------------------- | ----------------------------------------- |
| 13.1 Session State        | ✅ 100% — 11 字段全部对齐                 |
| 13.2 Runtime Event        | ⚠️ 55% — 文件路径/行数精度/timestamp 差异 |

3. Todo Analysis Result 因 LLM→关键词降级导致质量维度字段全部缺失
