## 14.1 没有 TodoWrite

**设计要求**: 整个 session 没有 TodoWrite 时，依赖 prompt 先验 + runtime score，仍然应完成一次路由决策并在合适时间锁定。

**实现状态**: ✅ 对齐

| 检查点                  | 状态 | 证据                                                                |
| ----------------------- | ---- | ------------------------------------------------------------------- |
| Prompt 先验始终执行     | ✅   | `decide()` 在每次 prompt 时运行，不依赖 TodoWrite                   |
| Runtime score 独立累积  | ✅   | `RuntimeTracker.track()` 对所有工具都触发，不依赖 TodoWrite         |
| 无 TodoWrite 时仍可锁定 | ✅   | `RUNTIME_ACCUMULATING + "threshold_reached" → LOCKED` 转移边        |
| 不阻塞路由              | ✅   | `maybe_redecide()` 中 `todowrite_signal` 参数为 Optional，可为 None |

**验证**: 转移表中有独立路径：`RUNTIME_ACCUMULATING → LOCKED`（threshold_reached），不经过 TODOWRITE_SIGNAL。

## 14.2 早期信息过少

**设计要求**: prompt 模糊 + runtime 不足 → 默认中低配启动 + 提高升级敏感度 + 让 Runtime 尽快触发升级。

**实现状态**: ✅ 对齐

| 检查点               | 状态 | 证据                                                       |
| -------------------- | ---- | ---------------------------------------------------------- |
| 模糊 prompt 中配启动 | ✅   | `_apply_conservative_bias()` 强制 ≥ medium                 |
| 升级敏感度高         | ✅   | 中等起始分（31）+ 低阈值（70），仅需 39 分增量即可 complex |
| Runtime 尽快触发     | ✅   | PostToolUse hook 每次工具调用都累积，信号密集              |
| 不降级               | ✅   | locked=False 首次决策可改，但只能向上                      |

---

## 14.3 频繁切换风险

**设计要求**: 明确禁止在同一 prompt 内反复切模型。原因：增加复杂度、破坏上下文连续性、带来不稳定收益。

**实现状态**: ✅ 对齐

| 检查点       | 状态 | 证据                                                                           |
| ------------ | ---- | ------------------------------------------------------------------------------ |
| 禁止反复切换 | ✅   | `DecisionRecord.locked=True` 后 `maybe_redecide()` 直接返回 None               |
| 一次升级窗口 | ✅   | 仅在 locked=False 期间可升级（首次 `decide()` 到 `maybe_redecide()` 锁定之间） |
| 锁定后不可逆 | ✅   | 状态机 LOCKED→COMPLETED 无出边回到 RUNTIME_ACCUMULATING                        |
| 无降级逻辑   | ✅   | `_max_complexity()` 只取较高者 + `_COMPLEXITY_RANK` 比较                       |

## 14.4 错误分类

**设计要求**: 即便初始分类错误，也允许在早期通过 Runtime 或 TodoWrite 纠正一次，但纠正后立即锁定。

**实现状态**: ✅ 对齐

| 检查点             | 状态 | 证据                                                                |
| ------------------ | ---- | ------------------------------------------------------------------- |
| 初始分类可纠正     | ✅   | `decide()` 返回 `locked=False`，注释 "首次决策可改"                 |
| Runtime 纠正路径   | ✅   | `maybe_redecide()` 中 runtime_score 触发升级                        |
| TodoWrite 纠正路径 | ✅   | `maybe_redecide()` 中 `todowrite_signal.is_implementation` 强制升级 |
| 纠正后立即锁       | ✅   | `promoted=True` 或 `need_lock=True` 时写 `locked=True`              |
| 纠正机会仅一次     | ✅   | locked 后 `maybe_redecide()` 返回 None                              |

**纠正窗口示意**:

```
decide() [locked=False]
  → track() × N 次 [locked=False, 可累积]
  → maybe_redecide() [升级 + locked=True]
  → 后续所有 maybe_redecide() 返回 None [locked]
```
