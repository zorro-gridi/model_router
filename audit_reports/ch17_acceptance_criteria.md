## 17.2 逐条审计

### V17-1 简单任务 token 消耗下降

- **机制状态**：✅ MiniMax-M3 作为 5/7 stage 的默认主模型
- **数据验证**：❌ 缺少历史 V0（关键词路由）数据对比
- **建议**：
  - 从 `stage_router.log` 提取 V1 上线后的 token 消耗
  - 与 V0 时期（如果有备份日志）的数据对比
- **风险等级**：P3（需数据回放）

### V17-2 中等任务返工率下降

- **机制状态**：⚠️ Workflow Planner（medium → double workflow）已实现，但当前几乎不触发（D10-4）
- **数据验证**：❌ `retry_count` 字段在 metrics 中未统计
- **建议**：
  - 增加 retry_count 指标（D15-1 同源）
  - 修复 workflow 触发逻辑（D10-4）后，返工率才有意义
- **风险等级**：P2

### V17-3 复杂任务强模型三步走

- **设计要求**：complex → 强模型规划 + 常规模型执行 + 强模型审计
- **当前状态**：❌ **未达成**
- **根因**：
  - D10-4：workflow 类型由 `is_simple` 决定，complex 几乎不触发
  - D10-5：`strong_model` 选择错误（implement stage fb = flash）
- **影响**：复杂任务（如跨模块重构）仍走单步或双步
- **修复路径**：
  1. 新增独立 `STRONG_MODEL = "deepseek-v4-pro"`
  2. 修复 `build_workflow_plan(complexity="complex")` 真正返回 triple
  3. 在 `proxy.do_POST` 中按 complexity 决定 workflow 类型
- **风险等级**：**P0**（核心价值未达成）

### V17-4 测试任务识别两类复杂度

- **设计要求**：
  - "写测试" → medium（用例生成）
  - "分析测试结果" → complex（根因定位）
- **当前状态**：❌ **未达成**
- **根因**：
  - D7-2：缺 `test` stage
  - D6.3-1：缺 stage 关键词 "测试 / 用例 / 覆盖率 / 失败 / 回归"
- **影响**：用户输入"帮我跑测试"或"分析测试失败"被错误路由到 audit / default
- **修复路径**：
  1. 补全 `test` stage（与 D7-2 同步）
  2. stage_detector.STAGE_KEYWORDS 补 "测试" / "用例" / "覆盖率" / "失败" / "回归" / "analyze test" / "test result"
  3. LLM classifier prompt 补 test stage 描述
- **风险等级**：**P0**（测试任务量大，路由错误会污染 audit 统计）

### V17-5 多窗口并发不污染

- **机制状态**：✅ state_index.json + per-session 文件 + 4 级查找
- **风险点**：D13-1（Level 3 timestamp 查找未实现）— 多 session 跨 project 时可能选错
- **建议**：实现 D13-1 的 4 级查找完整逻辑
- **风险等级**：P2

### V17-6 人工覆盖命令生效

- **机制状态**：✅ 7 个指令全部实现
- **缺陷**：
  - D12-2：`~stage explore` / `~stage test` 不识别（缺 stage）
  - D12-1：`~batch` 不强制流程起点
- **建议**：与 §7 / §5 修复同步
- **风险等级**：P1
