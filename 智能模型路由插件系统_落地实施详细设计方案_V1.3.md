# Claude Code 任务复杂度智能路由系统设计文档v1.3

## 1. 设计目标

本方案的目标不是“精确识别 Claude Code 的内部状态”，而是基于**可观测信号**，在**任务早期**尽快判断当前用户请求的整体复杂度，并完成一次稳定的模型路由决策。

核心目标有四个：

1. **放弃不可观测的 Stage 路由**，不再尝试推断 Explore / Implement / Review 这类内部状态。
2. **尽早完成复杂度定性**，在冷启动和早期运行阶段尽快决定本次 session 使用低配还是高配模型。
3. **偏保守路由**，宁可稍早升级，也不要让复杂任务长期停留在低配模型里。
4. **一次决策，整段锁定**，本次用户 prompt 对应的路由结果一旦确定，在下一次 prompt 提交前不再变化。

---

## 2. 总体结论

优化后的系统只保留两层核心判断：

- **Task Pattern**：任务模式分类，用于识别这次请求属于哪类工作。
- **Task Complexity**：任务复杂度分类，用于决定模型路由。

不再保留 Stage 分类，不再围绕“当前处于哪个执行阶段”做路由。

整体路由逻辑也不再是动态反复切换，而是：

**Prompt 先验判断 → 早期 Runtime 信号累积 → 首次 TodoWrite 深度判定 → 路由锁定**

---

## 3. 设计原则

### 3.1 放弃 Stage 路由

Stage 是 Claude Code 的内部认知状态，不是稳定的外部可观测量。代理层或 hook 层真正能拿到的是：

- 用户 prompt
- tool call 的名称
- tool call 的参数
- tool result
- 模型输出文本
- TodoWrite 列表

因此，Stage 不作为路由输入，只作为内部执行现象参考。

### 3.2 只做早期决策，不做后期反复切换

复杂度判断最有价值的窗口出现在 session 开始后的前几轮。越晚升级，省钱意义越小，且低配模型早期探索可能会污染后续判断。

所以系统采用：

- **早期尽快判断**
- **判断后立即锁定**
- **不做降级**

### 3.3 保守优先，升级门槛偏低

复杂任务误判为低复杂度的代价，远大于简单任务误判为高复杂度的代价。

因此路由策略应该偏向：

- 模糊任务：宁可略早升级
- 高风险任务：直接进入高配
- 低复杂度任务：才允许低配执行

### 3.4 权重支持自定义配置

提供不同 tool call、不同文件类型、不同文件行数对应的权重模版，允许由用户外部自定义配置，以适应用户不同的项目需求。

---

## 4. 系统架构

系统由五个核心模块组成：

### 4.1 Prompt Complexity Classifier

在 session 冷启动时，使用特定的 llm classifier 对用户 prompt 做一次轻量分类。

用途：

- 判断是否已经从 prompt 中直接暴露出高复杂度信号
- 给出初始模型建议
- 为后续 Runtime 分析提供先验

它不需要是重型 LLM 推理器，可以是：

- 规则匹配
- 关键词 + 模板
- 轻量 LLM 评估
- 规则与小模型混合

### 4.2 Runtime Complexity Tracker

基于项目配置的 tool call 权重体系持续累积 Runtime Complexity Score。

输入信号包括：

- tool 名称
- tool 参数
- tool result
- 文件路径
- 文件行数
- grep / search 返回规模
- bash exit code
- 是否出现错误、测试失败、重复编辑等

它是整个系统的主要动态信号源。

### 4.3 TodoWrite Analyzer

当 Claude Code 第一次产生完整的 TodoWrite 任务列表时，触发一次更强的复杂度分析。伪代码：

```python
if tool_name == "TodoWrite" and is_first_todo_write:
    complexity = llm_analyze_todos(todos)
    if complexity == "COMPLEX":
        # 立刻升级，不等 score 累积
        force_upgrade()
```

用途：

- 识别任务规划的整体规模
- 识别任务是否已经从探索进入明确执行
- 综合确认当前 session 的真实复杂度

这是本系统最关键的“早期强信号”。

### 4.4 Model Router

综合 Prompt 先验、Runtime Score、TodoWrite 分析结果，做最终路由决定。

session 初始化启动：

- 默认低配模型

路由目标：

- 低复杂度任务：低配模型
- 中复杂度任务：中配模型
- 高复杂度任务：高配模型

### 4.5 Decision Lock / Session State

一旦当前 prompt 的路由结果被最终确认，就写入 session state。

之后本次 prompt 的后续请求：

- 不再重新计算模型路由
- 不再允许降级
- 除非下一次新 prompt 到来，否则保持不变

## 5. 任务分类体系

本系统只保留两类分类结果。

### 5.1 Task Pattern

用于识别任务类型，典型包括：

- `explore`：探索与调研
- `architecture`：架构设计
- `feature`：新功能需求
- `audit`：审计系统功能
- `implement`：功能实现
- `debug`：调试异常
- `refactor`：模块重构
- `test`：测试相关
- `research`：调查研究
- `migration`：模块迁移
- `docs`：文档处理
- `ops`：运维、脚本、配置类任务

Task Pattern 的作用不是直接决定模型，而是辅助判断复杂度来源。

### 5.2 Task Complexity

用于最终路由的核心类别：

- `simple`
- `medium`
- `complex`

建议不要再拆得过细，否则会增加路由复杂度和误判成本。

---

## 6. 路由流程

## 6.1 冷启动阶段：Prompt 先验判断

当用户第一次提交 prompt 时，系统先做一次轻量分类。

输出内容：

- Task Pattern
- 初始 Complexity
- 初始模型建议
- 置信度

规则：

- 明确复杂任务：直接偏向高配
- 明确简单任务：低配启动
- 模糊任务：默认低配或中配，但设置更低的升级门槛

这里的关键不是“判断是否绝对正确”，而是快速形成一个可执行的初始路由。

---

## 6.2 早期运行阶段：Runtime Score 累积

session 启动后，系统通过 tool call 持续增加 Runtime Complexity Score。

重点不是工具数量本身，而是：

- 工具类型
- 工具输出规模
- 是否失败
- 是否触发大范围搜索
- 是否访问核心业务文件
- 是否发生重复修改和反复验证

这个阶段的目标是尽快让系统识别出“这不是简单任务”。

---

## 6.3 第一次 TodoWrite 阶段：最终复杂度确认

当 Claude Code 第一次输出完整 TodoWrite 列表时，系统触发一次更高权重的 LLM 分析。

分析内容包括：

- todo 数量
- todo 之间的依赖关系
- 是否涉及跨文件修改
- 是否包含测试、迁移、回归验证
- 任务拆分是否显示高复杂度
- 与已有 Runtime Score 是否一致

这里做最终路由确认。

---

## 6.4 决策锁定

一旦完成复杂度定性：

- 路由结果写入 session state
- 当前 prompt 的模型固定
- 后续请求不再切换
- 不做降级

这意味着，本次 prompt 只会经历一次路由收敛。

---

## 7. Runtime Complexity Score 设计

## 7.1 评分原则

Runtime Score 不是简单统计 tool call 数量，而是对不同信号赋予不同权重的体系设计。

权重配置体系包括：

- 按 claude code 常见的 toolname 分配权重，例如：read/write/grep/update/bash/search...等
- 按 tool 处理的文件后缀分配权重，例如：_.md 文件一般为文档，权重低;_.json 一般为配置记录，权重低;\*.py为主项目文件，权重高
- 按项目子目录分配权重，有的子目录为核心模块，权重天然高；有的为配置模块，权重较低

权重配置文件：

- 推荐 YAML 格式

原则如下：

- **读核心文件** 比读说明文件更重要
- **大范围 grep/search** 比单次读取更能说明复杂度
- **测试失败** 比普通执行更重要
- **编辑核心逻辑文件** 比编辑配置文件更重要
- **重复 read / edit / test 循环** 说明复杂度持续上升

---

## 7.2 Tool Score 基本规则

建议内置一套固定权重，并允许用户自定义。

示例规则：

- `Read README / docs`：低权重
- `Read 核心业务文件`：中高权重
- `Grep / Search 大范围命中`：高权重
- `Bash pytest / test`：中高权重
- `Bash 非零退出码`：高权重
- `Edit 核心业务文件`：高权重
- `TodoWrite 首次出现`：极高权重
- `重复失败 + 重试`：持续加权

---

## 7.3 文件权重规则

文件本身也应影响分值。

### 文件类型分层

不同文件类型有不同权重，例如：

- 核心业务逻辑文件：高权重
- 模型训练 / 算法核心文件：高权重
- 控制器 / 编排文件：中高权重
- 测试文件：中权重
- 配置文件：低到中权重
- 文档文件：低权重
- 生成文件：极低权重

### 行数分层

文件行数也影响复杂度：

- 小文件：低权重
- 中等文件：中权重
- 大文件：高权重
- 超大文件：更高权重

### 组合规则

例如：

- 读一个大核心文件，比读多个小 docs 文件权重大得多
- 读核心文件后又改同一核心文件，复杂度增长更快
- 多文件联动修改，复杂度会显著抬升

---

## 7.4 Runtime Score 不做降级判定

Score 在同一个 session 内只能增长，不会归零。

因此它的用途是：

- 尽快触发升级
- 尽快确认复杂任务
- 兜底识别未被 prompt 和 TodoWrite 明确捕获的复杂性

它不是用来做反向降级的。

---

## 8. Prompt 先验分类策略

Prompt 先验分类只做一件事：在冷启动时提供初始复杂度判断。

建议策略：

### 8.1 明确高复杂度请求

例如：

- 迁移整个支付系统
- 多模块联动重构
- 大范围测试修复
- 跨文件架构改造
- 新功能从 0 到 1 全链路实现

这类任务应直接偏向高配。

### 8.2 明确低复杂度请求

例如：

- 小范围修 bug
- 单文件文案修改
- 简单脚本修正
- 局部格式整理

这类任务可以先低配。

### 8.3 模糊请求

例如：

- “帮我修一下这个问题”
- “看下为什么报错”
- “帮我优化一下逻辑”

这类任务不应依赖 prompt 本身做强判断，而是让 Runtime 和 TodoWrite 尽快接管。

---

## 9. TodoWrite 分析策略

TodoWrite 是本系统非常重要的早期强信号。

### 9.1 为什么 TodoWrite 权重高

因为它往往出现在：

- 已完成初步理解之后
- 已形成执行规划之后
- 即将进入实质修改之前

这时系统已经拥有足够上下文，可以更准确识别任务整体复杂度。

### 9.2 TodoWrite 分析关注点

LLM 分析 TodoWrite 时，重点看：

- todo 数量
- 每个 todo 是否独立
- 是否跨文件
- 是否跨模块
- 是否包含回归测试
- 是否包含迁移 / 兼容 / 重构
- 是否存在大量依赖前置步骤

### 9.3 TodoWrite 触发后的策略

第一次 TodoWrite 出现时：

1. 立即分析
2. 与 Runtime Score 融合
3. 生成最终复杂度
4. 锁定模型

这一步不应拖延。

---

## 10. 模型路由策略

## 10.1 路由目标

路由不是追求完美，而是追求：

- 尽早识别复杂任务
- 尽量少让复杂任务在低配模型上耗损
- 保证简单任务成本可控

## 10.2 路由建议

可采用如下三档：

- `simple` → 低配模型
- `medium` → 中配模型
- `complex` → 高配模型

## 10.3 保守偏置

默认偏置建议如下：

- 模糊情况宁可往高配靠
- 复杂信号只要连续出现，就尽早升级
- 不要等到“百分百确认”才升级
- 升级后不再回退

---

## 11. Context Summary 注入

当模型从低配升级到高配时，需要注入一段简洁的上下文摘要。

### 11.1 目的

避免高配模型接手时完全从零理解前文，减少“别人开过头”的断裂感。

### 11.2 摘要内容

建议包括：

- 当前任务目标
- 已读文件数量和类型
- 已发生的关键编辑
- 测试结果
- 当前推断的任务复杂度
- 已完成的里程碑

### 11.3 注入时机

只在复杂度从低档跃迁到高档那一刻注入一次。

之后固定模型，不重复注入。

---

## 12. 事件状态机

可以把系统状态理解为下面这条链路：

### 12.1 初始态

`Idle → PromptReceived`

### 12.2 先验判定

`PromptReceived → Preclassified`

### 12.3 运行累积

`Preclassified → RuntimeAccumulating`

### 12.4 黄金信号触发

`RuntimeAccumulating → TodoTriggered`

### 12.5 最终决策

`TodoTriggered → FinalRouted`

### 12.6 锁定

`FinalRouted → Locked`

锁定后直到下一次 prompt 才重新开始。

## 14. 异常处理与边界条件

### 14.1 没有 TodoWrite

如果整个 session 没有出现 TodoWrite，就只能依赖：

- prompt 先验
- runtime score

这种情况下仍然应完成一次路由决策，并在合适的时间锁定。

### 14.2 早期信息过少

如果 prompt 模糊、runtime 也不足，则默认采取保守策略：

- 先中低配启动
- 提高升级敏感度
- 让 Runtime 尽快触发升级

### 14.3 频繁切换风险

本方案明确禁止在同一 prompt 内反复切模型。

原因：

- 增加复杂度
- 破坏上下文连续性
- 带来不稳定收益

### 14.4 错误分类

即便初始分类错误，也允许在早期通过 Runtime 或 TodoWrite 纠正一次，但纠正后立即锁定。

---

## 15. 补充的优化点

### 15.1 引入“首次强信号优先”原则

在多个信号同时出现时，优先使用最早的高置信度强信号，而不是等所有信号都收集完。

### 15.2 给复杂任务设置更低的升级阈值

不要追求高准确率后再升级，而是追求在第 1 次明显复杂信号出现时就升级。

### 15.3 高风险任务直通高配

对于迁移、重构、支付、测试修复、跨模块改造等高风险任务，可以设置更激进的直通规则。

### 15.4 记录路由理由

每次 final route 记录一句简短理由，便于后续调参：

- prompt 显式高复杂
- runtime score 快速上升
- TodoWrite 表现出多步骤依赖
- 测试失败后升级

### 15.5 只在 session 级别锁定，不做跨 session 继承

下一次 prompt 重新判断，不把上一次 session 的 score 带入新任务，避免误污染。

## 16. 最终推荐实现形态

### 推荐系统形态

1. **Prompt 先验分类**
2. **早期 Runtime Score 累积**
3. **第一次 TodoWrite 触发最终判定**
4. **注入一次上下文摘要**
5. **模型锁定直到下一次 prompt**

### 推荐保留项

- Task Pattern
- Task Complexity
- Runtime Complexity Score
- TodoWrite Analyzer
- Context Summary Injector
- Decision Lock

### 推荐删除项

- Stage Router
- 降级逻辑
- 外部可配置评分项
- 反复动态切模型

---

## 17. 最终结论

这套优化后的系统，本质上已经从“试图追踪 Claude Code 的内部阶段”，转变为“利用早期可观测信号，尽快完成一次稳定的任务复杂度定性”。

它的优势是：

- 逻辑自洽
- 可实现
- 可调参
- 对复杂任务更安全
- 对 session 连贯性更友好

最重要的是，它符合真实工程里的损失函数：
**复杂任务发现晚的代价，远高于简单任务误升的代价。**

## 18. `route_model` 字段更新链路（statusline 权威源）

`route_model` 是 `model_router_state_<sid>.json` 中**反映最近一次代理请求实际路由到的最终模型**的字段，是 `statusline.sh` 第三行直接读取的权威源。该字段由 6 个写入点协同维护，保证文件创建后永不缺失、永不滞后。

### 18.1 写入点总览

| #   | 触发时机                              | 模块                             | 写入策略                                 | 代码位置                                          |
| --- | ------------------------------------- | -------------------------------- | ---------------------------------------- | ------------------------------------------------- |
| 1   | 文件**首次创建**                      | `state_persistence.py`           | `route_model: null`（占位）              | `state_persistence.py:120-122`                    |
| 2   | 首个 `UserPromptSubmit`               | `stage_detector.py`              | **初始化** `init_route_model`            | `stage_detector.py:896-912`                       |
| 3   | `proxy.py do_POST` 主路径             | `proxy.py`                       | **覆写** `actual_route_model`            | `proxy.py:1734,1858-1864`                         |
| 4   | `proxy.py do_POST` fallback 成功      | `proxy.py`                       | **覆写** `actual_route_model = fb_model` | `proxy.py:1771`                                   |
| 5   | 用户 `~model <name>`                  | `stage_detector.py` + `proxy.py` | 双路径生效                               | `stage_detector.py:864-866`、`proxy.py:1514-1538` |
| 6   | 其它 `write()`（`maybe_redecide` 等） | `decision_engine.py`             | **字段继承**保留旧值                     | `state_persistence.py:138-144`                    |

### 18.2 初始化优先级链

`stage_detector.py` 的 `init_route_model` 表达式（**永不为 null**）：

```
1. new_model                          # 用户 ~model 显式覆盖（最高）
2. decision.final_model               # LLM 分类器决策
3. STAGE_CONFIG[resolved_stage].model  # 阶段默认主模型
4. "MiniMax-M3"                       # 硬编码兜底
```

### 18.3 实际路由态追踪

`proxy.py` 在每次请求后用 `actual_route_model` 写回：

- **sticky swap 后**：`actual_route_model = model`（主备已交换 → 实际是 fallback）
- **fallback retry 成功**：`actual_route_model = fb_model`
- 两者均通过 `SessionStateStore.write(route_model=actual_route_model, ...)` 写回

### 18.4 字段继承机制（兜底保护）

`state_persistence.py` 的 `optional_fields` 循环实现**写时继承**——`write()` 调用方未传 `route_model` 时**自动从 existing 继承**，避免误清空：

```python
for key in optional_fields:
    if key in kwargs and kwargs[key] is not None:    # 显式传 → 用新值
        new_data[key] = kwargs[key]
    elif key in existing:                            # 未传 → 从旧文件继承
        new_data[key] = existing[key]
```

### 18.5 ~model 命令的双路径生效

- **路径 A（stage_detector）**：解析 `~model <name>` → `new_model` → 写入 `route_model=new_model`
- **路径 B（proxy）**：从请求 body 解析 `prompt_model_override` → 直接路由 → 写回 `route_model=actual_route_model`

两条路径确保用户在 `~model ds-v4-pro` 的**当前回合**就立即生效。

### 18.6 ~model reset 行为

2026-06-16 行为变更后，`~model reset` 同时承担两件事：

1. **解除本回合的 ~model 一次性覆盖**：`prompt_is_reset=True` → `model_override = None`，本请求回到 stage 路由
2. **清除 sticky fallback**：`clear_fallback()` → 删除 `fallback_<sid>` 文件，让后续请求回到正常 stage 路由

`stage_detector` 端对 `~model reset` 保持 no-op（与 "~model 一次性" 语义对齐）；但 proxy 端仍会执行 `clear_fallback()`，因为 sticky fallback 是**模型不可用时自动积累的副作用**，需要用户显式清除。

### 18.7 时序保证

```
UserPromptSubmit (stage_detector)     ──→  创建文件 + 初始化 route_model
        ↓
proxy do_POST (实际请求)              ──→  用 actual_route_model 覆写
        ↓
PostToolUse (maybe_redecide)         ──→  只读 + 字段继承，不创建也不清空
```

- stage_detector **先于** proxy 第一次请求 → 文件必先存在
- `maybe_redecide` 在 `decision_engine.py:212` 有 `if not state: return None` 保护 → **不会先于 stage_detector 创建文件**
- 一旦文件存在，**任何** `write()` 调用都不会把 `route_model` 写回 null
